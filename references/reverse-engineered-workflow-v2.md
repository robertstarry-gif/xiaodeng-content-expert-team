# Reverse-Engineered Editorial Workflow V2

This public workflow is based on observing a staged content team process. It does not copy private roles or content; it generalizes the mechanism.

## What Works

The stronger workflow is not "many experts in one answer." It is a sequence of separate editorial jobs with handoffs.

1. **Lead Editor clarifies direction**
   - If the user rejects the old direction, ask for new style, audience, and length.
   - Turn the answer into a new article positioning and outline.
   - Define evaluation dimensions before research or drafting.

2. **Source Scout gathers usable material**
   - The scout returns structured facts, examples, prices, user reactions, scenarios, scoring suggestions, and common misunderstandings.
   - The scout does not write the article.
   - The report gives the writer raw material in reader-facing language.

3. **Draft Writer writes from constraints**
   - The writer receives original material, target style, required structure, source report, and concrete constraints.
   - The writer changes format and angle, not just wording.
   - A strong draft uses simple metaphors, tables, scenario labels, pros/cons, and "avoid pitfall" notes when appropriate.

4. **Skeptical Reviewer audits after a draft exists**
   - The reviewer checks consistency, factual risk, naming, parameters, structure, and reader fit.
   - Good review produces exact fixes, not vague taste comments.

5. **Lead Editor creates a revision ticket**
   - The lead editor converts review into numbered edits.
   - Each edit says what to change and why.
   - The writer revises the artifact, not the explanation.

6. **Verification before delivery**
   - The lead checks each revision item.
   - The final response summarizes what changed and delivers the artifact.

## Why This Reduces AI Feel

AI-feel usually survives when the same pass has to invent angle, gather facts, write, judge, and polish at once. The staged workflow forces tension:

- research has to provide concrete material;
- the writer must write for a defined reader;
- the reviewer is allowed to attack the draft;
- revision is driven by specific problems.

## Content Moves Worth Preserving

- Ask one direction question when the user's dissatisfaction is broad.
- Build a scoring or evaluation system before comparison writing.
- Use "说人话" descriptions for ordinary readers.
- Use concrete misunderstanding sections for beginner content.
- Use tables only when they make decisions easier.
- Use a revision ticket with numbered fixes.
- Verify each fix before delivery.

## Failure Modes To Avoid

- Showing the user the whole internal transcript by default.
- Letting the writer invent unsupported facts.
- Treating a style complaint as a request for more polish.
- Using a metaphor because it is cute rather than because it explains.
- Writing final slogans that feel like generated motivation.
- Calling a draft "human" before a skeptical review has attacked it.
