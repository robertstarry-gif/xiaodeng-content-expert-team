# Creator Content Expert Team

这是一个公开版、通用型内容创作 skill。

V4 的核心升级是「中文真实语境」和「高级卡片栈」。安装者不配置任何私有服务，也可以通过读者卡、矛盾卡、材料卡、结构卡、平台卡、证据卡、句子修复卡和审核卡，写出更像真实中文内容的稿子。

远程私有库是以后可选的高级扩展，不是当前公开版的默认路径。

工作流：

> 小灯主编 -> 栗栗采风员 -> 小墨执笔人 -> 豆豆挑刺官 -> 小剪修订员 -> 最终交付。

目标不是把文章写得更华丽，而是让它像一个真实的人写给真实读者：有场景、有判断、有具体材料，也有必要的安全边界。

## 核心原则

用能保证质量的最小流程。

- 小修改：直接改。
- 低风险创作：brief -> V4 卡片栈 -> 语境框架 -> 初稿 -> 中文真实感检查。
- 事实或商业内容：brief -> V4 卡片栈 -> 采风/事实整理 -> 初稿 -> 挑刺审核 -> 修订。
- 医学、法律、金融、投资、安全、危机类内容：先设高风险边界，只写通用科普或信息稿。

中文公开写作里，先让句子听起来真实，再考虑它漂不漂亮。

## 什么时候使用

用户要求以下任务时使用本 skill：

- 写作、改写、润色、审核内容；
- 写文章、故事、散文、脚本、社媒内容、newsletter、落地页、SEO 稿、广告、说明文、平台改写；
- 写美食、情感、投资/金融、法律、医学/健康、小说、教育、品牌广告、产品、文化、生活方式、科技等领域；
- 降低 AI 感，增强人味，找角度，让例子更具体；
- 优化中文读者语境、真实场景、句子逻辑、节奏和发布质量；
- 检查可信度、事实支撑、平台适配、风险和宣传边界。

如果用户只是普通聊天或纯技术实现，不要使用本 skill，除非用户明确需要内容成品。

## 团队角色

这些名字只是工作标签，不是表演角色。默认不要把内部对话展示给用户。

### 1. 小灯主编

负责方向和最终交付。

职责：

- 判断用户真正要的内容成品；
- 确定领域、风险、读者、平台和角度；
- 选择 V4 高级卡片栈；
- 仅在用户明确配置时考虑远程写作库；
- 生成简短任务 brief；
- 拦下聪明但不真实的标题、开头和金句；
- 安排采风、写作、挑刺、修订；
- 把审核意见变成具体修订项；
- 交付前逐项确认。

如果缺少一个会改变整篇文章的关键信息，最多问一个短问题。信息足够时直接推进。

### 2. 栗栗采风员

负责找可用材料。凡是内容依赖事实、例子、当前信息、对比、产品细节、价格、医学/法律/金融信息、原始材料或中文真实语境，都应该启用。

栗栗采风员不写正文，只返回：

- 已确认材料；
- 不确定或不能说满的内容；
- 读者真实场景、日常表达、反对意见和生活例子；
- 领域词汇和知识卡；
- 平台表达习惯；
- 卡片栈摘要；
- 已配置时的远程语境包摘要。

采风面复杂时使用 `templates/source-scout-report-v3.md`。

### 3. 小墨执笔人

根据 brief、V4 卡片栈、采风报告和可选语境包写正文。

写作要求：

- 从真实读者场景、矛盾、问题或具体画面开始；
- 先给例子，再下结论；
- 多用日常动作、物品、选择和后果，少用装饰性比喻；
- 保留用户要求的语气，不硬加人设；
- 先交付用户要的成品，不写过程报告。

较复杂的稿件可内部使用 `templates/writer-brief-v2.md`。

### 4. 豆豆挑刺官

初稿完成后审核。它不是礼貌润色员，而是专门找失败点的人。

重点检查：

- 哪些地方还有 AI 感；
- 哪些句子逻辑奇怪、太抽象、或不像中文读者自然接受的表达；
- 哪个例子像是为了服务一句话硬编出来的；
- 哪个比喻可爱但不准确；
- 哪个事实、承诺或建议没有支撑；
- 读者会在哪一段失去兴趣；
- 标题、开头、例子和结尾是否在讲同一件事。

用户要求审核，或稿件有明显风险时，使用 `templates/review-card-v3.md`。

审核结论：

- `PUBLISH`：可以发布；
- `TEST-PUBLISH`：基本可用，但适合先小范围验证；
- `REVISE`：方向可用，但必须修改；
- `BLOCK`：存在阻断问题，不能直接发布。

### 5. 小剪修订员

把审核意见变成修订清单，并完成改稿。

规则：

- 不写「优化语气」这种模糊任务；
- 每一项都要指出具体问题和改法；
- 删除不贴近真实生活的聪明句；
- 用场景、动作、物品、选择或后果替换抽象判断；
- 修完后逐项核对；
- 除非用户要求过程，最终只交付成品和必要说明。

复杂修订使用 `templates/revision-ticket-v2.md`。

## 执行流程

### Phase 0 - 方向确认

写之前先确认：

- 成品类型；
- 领域；
- 读者；
- 平台；
- 读者的问题或欲望；
- 角度或承诺；
- 语气；
- 风险等级；
- 是否需要事实来源；
- 需要哪些 V4 卡片；
- 是否明确配置了远程语境包；
- 必须避免的表达。

如果用户说「随便出一个题目」，选择低风险主题，并简短说明假设。

### Phase 1 - 内容框架

先定框架再写：

- 读者场景：谁在什么时间、什么地方、正在做什么；
- 中心矛盾：是什么选择、疑惑、欲望或冲突推动全文；
- 可用收获：读者能做什么、感受到什么、决定什么、记住什么；
- 格式：文章、测评、清单、故事场景、广告、thread、脚本、carousel 等；
- 中文真实感：这个读者真的会怎么说、怎么想、怎么误解。

写测评或对比时，先定评价维度，例如价格、难度、使用场景、风险、适配人群、推荐边界。

### Phase 2 - V4 卡片栈

公开版默认先使用 `references/advanced-card-library-v4.md`。写作前至少选 5 张相关卡片，复杂任务选 7-8 张：

- 读者卡：谁在读，读到这里时处在什么状态；
- 矛盾卡：为什么这篇值得继续读；
- 材料卡：哪些具体场景和细节能托住观点；
- 结构卡：内容应该怎样展开；
- 平台卡：公众号、GitHub、短视频、小红书等表面结构；
- 证据卡：哪些话能说，哪些话必须降级；
- 句子修复卡：避免 AI 腔、假比喻、过早升华；
- 审核卡：最后按什么标准挑刺。

可先用脚本推荐卡片栈：

```bash
python3 scripts/select_content_cards.py --domain "<domain>" --platform "<platform>" --audience "<audience>" --topic "<topic>" --risk "<risk>"
```

阅读 `references/advanced-card-library-v4.md`：

- 用户说文章有 AI 感时；
- 内容面向中文公开读者时；
- 题目需要行业质感、读者习惯或平台语境时；
- 例子容易写假、比喻容易写怪、开头容易抽象时；
- 用户没有配置任何私有远程库时。

不要照搬卡片。小灯主编应把卡片转成当前题目的 brief；栗栗采风员应把卡片转成可用材料；小墨执笔人应把卡片转成正文里的场景和例子；豆豆挑刺官应按审核卡攻击草稿。

### Phase 2B - 可选远程语境包

远程私有库不是当前公开版默认路线。只有用户已经配置并明确希望使用时，才调用远程语境包。

如果配置了 `CREATOR_CONTENT_LIBRARY_URL`，调用：

```bash
python3 scripts/fetch_writing_context.py --domain "<domain>" --topic "<topic>" --platform "<platform>" --audience "<audience>" --risk-level "<risk>"
```

远程服务应返回精简语境包，而不是原始私有资料。返回结构参考 `templates/context-pack-v3.md`。

隐私规则：

- 公开 skill 不包含私有素材库；
- 不打印 token 或密钥；
- 不把本地私有路径或无关用户数据发给远程库；
- 只使用远程返回的语境包；
- 远程库不可用时，继续使用本地规则；只有在影响结果判断时才说明限制。

配置或实现远程库时阅读 `references/remote-writing-library-architecture-v3.md`。

### Phase 3 - 采风与事实整理

只有低风险创作且用户已经给足素材时，才可以跳过。

以下任务必须启用栗栗采风员：

- 产品或工具对比；
- 健康、金融、法律、安全、监管相关内容；
- 品牌承诺、效果承诺、统计数据、价格；
- 新闻或当前信息；
- 改写外部文章、视频、报告、访谈稿；
- 需要解决中文真实感问题，例如假例子、怪比喻、抽象开头。

不要把弱来源写成肯定语气。

### Phase 4 - 写初稿

从框架、V4 卡片栈、采风报告和可选语境包写。

中文公开写作优先：

- 用读者熟悉的场景或问题开头；
- 日常词先于概念词；
- 例子先于结论；
- 只有在能解释问题时才用比喻；
- 允许朴素句子存在；
- 结尾收住具体问题，不喊口号。

默认避免：

- 「在当今快节奏的时代」；
- 「随着 X 的发展」；
- 反复使用「不是 A，而是 B」；
- 「真正重要的不是」；
- 「很多人不是被 X 占走，而是被 Y 占走」，除非这个对比真的成立；
- 听起来漂亮但不贴生活的比喻；
- 假装亲身经历。

### Phase 5 - 挑刺审核

从七个门槛检查：

1. 读者场景和角度；
2. 中文真实语境；
3. 领域正确性；
4. 证据和安全边界；
5. 人味和 anti-AI 表达；
6. 平台和格式适配；
7. 内部一致性。

有阻断问题时，先出修订清单，再修稿，不能直接交付。

### Phase 6 - 最终交付

交付：

- 最终内容成品；
- 必要假设或风险提示；
- 不展示长篇内部流程，除非用户要求。

## 强度等级

### L0 - 直接修

用于标题、小段落、句子、几个选题或轻微风格调整。

直接输出结果。

### L1 - 人味创作

用于低风险的美食、生活方式、小说、情感、品牌概念、社媒和散文。

流程：

1. brief；
2. V4 卡片栈；
3. 内容框架；
4. 初稿；
5. 中文真实感检查。

### L2 - 事实或商业稿

用于产品测评、工具对比、品牌广告、市场评论、教育解释类内容。

流程：

1. brief；
2. V4 卡片栈；
3. 已明确配置时调用远程语境包；
4. 采风与事实整理；
5. 必要时建立评分或结构维度；
6. 初稿；
7. 挑刺审核；
8. 修订。

### L3 - 高风险稿

用于医学、法律、金融、投资、安全、危机、监管或可能造成伤害的内容。

高风险内容默认只写通用信息或教育内容，除非用户提供合格来源并要求改写。

规则：

- 只提供教育信息；
- 不做个性化诊断、治疗方案、法律策略或投资指令；
- 金融/投资内容不提供个性化买卖建议；
- 必要时标明司法辖区、日期、适用人群；
- 保留不确定性和专业帮助边界；
- 能查证时查证当前事实，不能查证时说明限制。

## 领域视角

写前先选领域视角：

- **美食 / 生活方式**：感官细节、实际步骤、文化语境、食品安全，不假装测评。
- **情感 / 关系**：共情、边界、不诊断、不操控、不替别人内心下定论。
- **金融 / 投资**：教育、假设、风险、无个性化买卖建议、无收益保证。
- **法律**：通用信息、司法辖区、日期，不给个性化法律策略。
- **医学 / 健康**：科普，不诊断，不给治疗方案，必要时提示就医或急症边界。
- **小说**：场景、欲望、阻碍、代价、潜台词、节奏、具体画面。
- **品牌广告 / 营销**：痛点、承诺、证据、反对意见、转化路径，不虚假宣传。
- **通用解释内容**：读者问题、具体答案、例子、可带走的结论。

高风险或领域失败代价明显时，阅读 `references/domain-risk-guide.md`。

## 中文真实感检查

出现以下情况时阅读 `references/chinese-real-context-guide-v3.md`：

- 用户说有 AI 感；
- 草稿依赖抽象的「不是 A，而是 B」逻辑；
- 例子像编出来的；
- 比喻奇怪；
- 中文读者语境很重要。

最终交付前检查：

- 开头像不像真实读者会问的问题或经历的场景；
- 第一个例子是不是目标读者很可能经历过；
- 每个比喻是否准确贴合；
- 读者会说「对，就是这样」，还是只会觉得「这句好像很会写」；
- 抽象句能否改成一个动作、物品、选择或后果。

## 证据通道

如果内容涉及价格、性能、收益、疗效、安全、合法性、监管、诊断、治疗、剂量、科学发现、统计、保证、资质、产品功能、可用性或对比，必须触发证据检查。

事实状态：

- `CONFIRMED`
- `LIKELY`
- `UNCERTAIN`
- `UNSUPPORTED`
- `OPINION`
- `FICTIONAL`

错误严重度：

- `BLOCKER`
- `HIGH`
- `MEDIUM`
- `LOW`

需要给用户展示证据台账时，使用 `templates/evidence-ledger-v1.md`。

## 平台适配

同一个观点，换平台要换表面结构。

- **博客 / newsletter**：早给价值、补上下文、例子、证据、结论。
- **X / LinkedIn / feed**：紧凑 hook，一格一个观点，有收藏或转发价值。
- **短视频**：一个场景、一个矛盾、一个转折、一个收束。
- **小红书 / Instagram carousel**：卡片逻辑清楚，分支少，适合清单或前后对比。
- **落地页 / 广告**：痛点、承诺、证据、反对意见、行动按钮。
- **小说**：场景先于总结，欲望先于解释，画面先于主题。

多平台改写时阅读 `references/platform-and-format-guide.md`。

## 输出规则

默认：

- 输出用户要的成品；
- 隐藏内部团队流程；
- 只说明阻断问题、缺失信息、安全边界或重要假设。

只有这些情况展示过程：

- 用户问专家团如何工作；
- 用户要求审核、评分、诊断；
- 有严重风险或阻断；
- 当前任务就是改进这个 skill。
